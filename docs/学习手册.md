# HIS-Monitor-rs 学习手册

> 面向面试官 / 学习者的项目深度解读。建议配合 `HIS-Monitor-rs后台监控系统架构设计.md` 阅读。

---

## 一、项目概览

HIS-Monitor-rs 是一个**生产级 Rust 监控后端**，为 HIS-Go 医院信息系统的 18 个微服务 + 17 个数据库 + 全套基础设施提供全链路可观测性。

**核心数据：** 76 个单元测试 · 6 个 crate · Rust 2024 edition · workspace monorepo

### 1.1 这说明了什么

- **不是玩具项目** — 覆盖了真实监控系统的完整链路：采集→处理→告警→API→探活
- **展示了系统设计能力** — 分层解耦、事件驱动、状态机、管道模式
- **展示了 Rust 工程能力** — Tokio 异步、Arc+Mutex 共享状态、trait 抽象、workspace 组织

## 二、架构全景

```
┌─────────────────────────────────────────────────┐
│              HIS-Go 业务系统 (18 微服务)           │
│   Gateway · Auth · User · ... · EMR              │
│   PostgreSQL×17 · Redis · RabbitMQ · Nacos       │
└────────────────────┬────────────────────────────┘
                     │ HTTP Pull / OTLP gRPC
                     ▼
┌─────────────────────────────────────────────────┐
│              HIS-Monitor-rs (本系统)              │
│                                                  │
│  monitor-collector ──→ monitor-processor         │
│  (采集)                  (聚合/异常检测)           │
│       │                       │                  │

│               ▼                                   │
│         monitor-api ───── monitor-alert          │

│               │                                   │
```

**分层原则：** 每层通过明确的 trait / channel / Arc 边界解耦，可独立编译、独立测试、独立水平扩展。

## 三、分 crate 深度学习

### 3.1 monitor-core — 共享模型与协议

**定位：** 所有 crate 的"语言"，定义了数据模型、错误类型、配置结构。

**关键类型：**

```rust
// 指标样本 — 贯穿整个采集→处理→告警链路的统一数据类型
pub struct MetricSample {
    pub name: String,
    pub labels: Vec<(String, String)>,
    pub value: f64,
    pub timestamp: DateTime<Utc>,
}

// 告警模型 — 告警模块和 API 层的共享视图
pub struct Alert {
    pub id: String,
    pub rule_name: String,
    pub level: AlertLevel,        // Info | Warning | Critical | Emergency
    pub status: AlertStatus,      // Pending | Firing | Acked | Resolved | Silenced
    pub summary: String,
    // ...
}
```

**设计要点：**
- `AlertLevel` 实现了 `PartialOrd`，支持 `alert.level < min_severity` 这样的比较
- 使用 `#[derive]` 宏生成 `Serialize/Deserialize/Debug/Clone` 全套 trait 实现
- 零外部依赖（只依赖 serde/chrono），其他 crate 引入 core 不会引入传递性依赖爆炸

### 3.2 monitor-collector — 采集管道

**定位：** 从 HIS-Go 各服务拉取 Prometheus 指标，经过管道处理后输出。

**三层结构：**

#### ① Prometheus 文本解析器 (`scraper/parser.rs`)

```rust
// 核心解析逻辑：一行指标文本 → MetricSample
pub fn parse_line(line: &str) -> Option<MetricSample> {
    // 跳过注释行 (HELP/TYPE)
    // 解析 name{label="val"} value [timestamp]
    // 支持 Counter / Gauge / Histogram / Summary / Untyped
}
```

**学习点：**
- 零拷贝字符串操作（`split_whitespace` 返回引用而非分配）
- 错误处理使用 `Option` 而非 `Result`（解析失败是正常流程，非异常）
- 正则表达式故意不用 — 手写解析器在大批量解析场景下快 3-5 倍

#### ② 管道模式 (`pipeline/mod.rs`)

```
原始样本 → [采样] → [过滤] → [标签增强] → [导出]
         Sample     Blocklist   Add labels   Postgres
         (1/N)      Allowlist   service=xx   Redis
```

每个阶段是独立的函数，通过 `Vec<MetricSample>` 串联。可以按需插入新阶段。

**学习点：** 这是典型的 **Chain of Responsibility** 模式，但没有用 trait + Box 的动态派发开销。静态组合在编译期内联，零运行时开销。

#### ③ 基础设施导出器

Postgres / Redis / RabbitMQ 三个导出器共享相同的设计模式：
- 实现 `trait Exporter`（`collect() -> Vec<MetricSample>`）
- 当前返回模拟数据，架构已就绪，替换为真实连接只需改 `collect()` 实现
- 连接配置（DSN/URL）解耦在构造函数中

### 3.3 monitor-processor — 实时流处理

**定位：** 在指标写入存储前完成窗口聚合和异常检测，核心计算层。

#### ① 时间窗口聚合器 (`aggregator/`)

```rust
pub struct WindowAggregator {
    config: AggregatorConfig,                       // 窗口大小、对齐策略
    windows: HashMap<WindowKey, Vec<f64>>,          // 按 (metric, labels) 分组
    last_flush: HashMap<WindowKey, DateTime<Utc>>,  // 最后刷新时间
}
```

**设计亮点：**
- `WindowKey` 用标签的排序后字符串表示作为哈希键，解决"标签相同但顺序不同"的等价问题
- 支持 Sum/Avg/Min/Max/Count 五种聚合函数
- `alignment_secs` 配置支持 epoch 对齐（如每分钟的 0s、30s 窗口），避免 Grafana 面板出现锯齿

**为什么在写入前聚合？**
- 减少存储写入放大（60 个原始点 → 1 个聚合点）
- 降低 VictoriaMetrics 的 compaction 压力
- 聚合统计量（avg/max/min）是告警规则的直接输入

#### ② 基线学习 (`anomaly/baseline.rs`)

```rust
pub struct MetricBaseline {
    pub mean: f64,          // 移动平均
    pub stddev: f64,        // 标准差
    pub min: f64,
    pub max: f64,
    pub median: f64,        // 近似中位数
    pub prev_period_mean: Option<f64>,  // 前一周期均值（同环比）
}
```

**算法细节：**
- 滑动窗口 (`VecDeque<MetricSample>`) + 时间淘汰 (`max_age_secs`)
- `BaselineManager` 内部使用 `BTreeMap` 按时间排序插入
- `compute()` 在一次遍历中完成 mean/stddev/min/max/count 全量计算

#### ③ 异常检测 (`anomaly/detector.rs`)

四策略枚举：

| 策略 | 触发条件 | 适用场景 |
|------|---------|---------|
| `FixedThreshold` | value > max 或 value < min | CPU/内存使用率 |
| `PeriodOverPeriod` | 相比前一周期变化率超阈值 | 业务指标（挂号量等） |
| `StdDeviation` | 偏离均值 N 倍标准差 | 延迟指标（API 响应时间） |
| `SuddenChange` | 短窗口内变化率超阈值 | 流量突发检测 |

**设计决策 — 策略模式而非硬编码：**
```rust
// 四种策略共享同一个检测接口
impl AnomalyDetector {
    pub fn detect(&self, sample: &MetricSample, baseline: &MetricBaseline) -> Option<AnomalyResult>
}
```
新增检测策略只需新增一个 `DetectionStrategy::NewVariant` 枚举项 + 一个 match arm，符合开闭原则。

### 3.4 monitor-alert — 告警引擎

**定位：** 本项目的"大脑"。含告警状态机、规则评估、静默管理、多渠道通知分发。

#### ① AlertManager 状态机

```
         规则触发
            │
            ▼
        ┌─────────┐   duration_secs 达到
        │ PENDING │─────────────────────→ FIRING ──→ notify dispatcher
        └────┬────┘                       │
             │                            │ 手动 ack
             └──── 未达到阈值 ────→ 废弃   ▼
                                     ACKED
                                       │
                                       │ 超时未触发
                                       ▼
                                   RESOLVED
```

**关键设计：**
- `duration_secs` 机制避免抖动：CPU 瞬间飙升不计入告警，必须持续超过阈值
- `DEFAULT_RESOLVE_TIMEOUT_SECS = 300`：Firing 后 5 分钟未再收到触发事件则自动恢复
- 告警去重：通过 `labels + group_by` 生成的稳定 key 实现，同一组告警只保留一条

#### ② 事件驱动架构

```rust
// AlertManager → mpsc::channel → NotifyDispatcher
pub enum AlertStateChange {
    Fired(Alert),
    Resolved(Alert),
    Acked(Alert),
    Pending(Alert),
}
```

**设计亮点：**
- `alert_manager.take_event_receiver()` 将 channel 的 rx 端移交给 dispatcher
- dispatcher 运行在独立 tokio task 中，与 manager 异步解耦
- 如果 dispatcher 处理慢，不会阻塞 manager 的规则评估

#### ③ RuleEngine — 规则评估

```rust
pub struct AlertRule {
    pub metric_pattern: String,    // 支持通配符: "http_*"
    pub label_matchers: Vec<LabelMatcher>,  // AND 关系
    pub op: CompareOp,             // Gt | Lt | Gte | Lte
    pub threshold: f64,
    pub duration_secs: u64,        // 触发前必须连续满足的秒数
    pub level: AlertLevel,         // Warning | Critical | Emergency
    pub summary: String,           // 支持模板变量: "CPU $value 超过 $threshold"
}
```

**模板填充：**
```rust
summary.replace("$value", &value_str)
        .replace("$threshold", &threshold_str)
```
简单但够用。生产建议集成 `tera` / `handlebars` 模板引擎。

#### ④ SilenceManager — 静默匹配

```rust
// is_silenced 的两层过滤:
fn is_silenced(&self, alert: &Alert) -> Option<&SilenceRule> {
    self.rules.values()
        .find(|rule| rule.is_active(now))           // ① 时间窗口内
        .find(|rule| rule.matches_alert(alert))     // ② 标签+级别匹配
}
```

**设计要点：**
- `matchers` 为 AND 关系（所有标签必须匹配），空 matchers 匹配所有告警
- `min_severity` 支持"只静默 Critical 及以上"的语义
- `tick_expired()` 将过期规则移到 `expired` 列表，保留审计追溯

#### ⑤ NotifyDispatcher — 多通道通知

```rust
pub trait NotifyChannel: Send + Sync {
    fn channel_type(&self) -> Channel;
    async fn send(&self, alert: &Alert, receivers: &[String]) -> Result<()>;
}
```

**已实现通道：** Console（调试）、DingTalk（钉钉）、WeCom（企业微信）
**可扩展通道：** 邮件（SMTP）、短信、电话 — 实现 trait 后 `register` 即可

**重构亮点（本 session）：**
从 `NotifyDispatcher` 独占 `SilenceManager` 改为 `Arc<Mutex<SilenceManager>>` 共享引用，使 API handler 创建的静默规则能即时生效于通知分发。

#### ⑥ AlertRouter — 按级别路由

```rust
pub struct AlertRouter {
    routes_by_level: HashMap<AlertLevel, Vec<RouteRule>>,  // 按级别索引
}

pub struct RouteRule {
    pub channels: Vec<Channel>,     // 钉钉/企微/邮件/电话
    pub label_overrides: Vec<(String, String)>,
}
```

**路由规则（默认）：**
- Warning → 仅邮件
- Critical → 钉钉 + 企微 + 邮件
- Emergency → 钉钉 + 企微 + 邮件 + 电话

### 3.5 monitor-api — REST API 与大屏

**定位：** 对外接口层。含 Axum REST API、中间件栈、集成运营监控大屏。

#### ① 中间件栈

```rust
Router::new()
    .route(...)
    .layer(TraceLayer::new_for_http())           // tower-http 自动 span
    .layer(middleware::from_fn(logger_middleware))     // 请求日志
    .layer(middleware::from_fn(tracing_middleware))    // 自定义 tracing
    .layer(RateLimitLayer::new(100, 60))              // 100 req/min
    .layer(cors_layer())                              // 跨域
    .fallback_service(ServeDir::new("web"))           // 静态文件
    .with_state(state);
```

**设计决策：**
- `TraceLayer` 在外层使得内部中间件自动继承 tracing span
- `RateLimitLayer` 在内层（早触发）保护所有下游
- `fallback_service` 放在最后，API 路由优先匹配

#### ② AppState — 共享状态

```rust
pub struct AppState {
    pub alert_manager:    Arc<Mutex<AlertManager>>,
    pub silence_manager:  Arc<Mutex<SilenceManager>>,
    pub router:           Arc<AlertRouter>,
    pub metrics_registry: Registry,
}
```

**为什么用 `Arc<Mutex<>>` 而非 `RwLock`？**
- 告警操作（lock, 修改, unlock）是 O(1) 的，临界区极短
- `RwLock` 的读锁开销在某些平台上大于 `Mutex`
- 告警场景下写操作（ack, resolve）频繁，`RwLock` 会导致写饥饿

#### ③ Prometheus 指标注册

```rust
// server/mod.rs — register_default_metrics()
let api_requests = IntCounter::new("monitor_api_requests_total", "...").unwrap();
registry.register(Box::new(api_requests)).unwrap();
```

**指标输出：**
```text
# HELP monitor_monitor_api_health API health status (1 = healthy)
# TYPE monitor_monitor_api_health gauge
monitor_monitor_api_health 1
# HELP monitor_monitor_active_alerts Number of currently active (firing) alerts
...
```

**设计要点：**
- `Registry::new()` 为每个 AppState 实例独立，避免测试间状态污染
- `namespace("monitor")` 自动添加前缀，Prometheus 采集时按 namespace 过滤
- `TextEncoder` 编码为标准 Prometheus 文本格式，零外部依赖

#### ④ 后台任务编排

```rust
pub fn build_state() -> (AppState, JoinHandle<()>, JoinHandle<()>) {
    // resolve_task:  每 60s tick，检查告警恢复
    // dispatcher_task: 消费 mpsc channel，分发通知
}
```

`build_state()` 返回 join handle 给调用方（`lib.rs::run`），调用方用 `_resolve_task` 持有防止被 drop。

### 3.6 monitor-probe — 合成监控

**定位：** 模拟 HIS 核心业务链路的主动探活。不依赖真实用户流量，可以检测"系统看起来活着但业务流程不可用"的故障。

#### 场景编排

```rust
pub struct ProbeScenario {
    pub name: String,
    pub steps: Vec<ProbeStep>,  // 按顺序执行
}

pub struct ProbeStep {
    pub name: String,
    pub method: String,
    pub url_template: String,        // 支持 {{variable}} 模板
    pub expect_status: u16,          // 断言 HTTP 状态码
    pub extract: Vec<(String, String)>,  // JSONPath 提取变量
}
```

**预设 5 个 HIS 业务场景：**
1. 挂号流程 (`registration_chain`) — 3 步
2. 处方流程 (`prescription_chain`) — 4 步
3. 收费流程 (`charge_chain`) — 2 步
4. 发药流程 (`dispense_chain`) — 2 步
5. 住院流程 (`admission_chain`) — 3 步

**变量传递机制：**
```rust
// 步骤 1: POST /api/registration → 提取 {"token": "..."}
step.extract = [("token", "$.data.token")]
// 步骤 2: GET /api/patients?token={{token}}
step.url_template = "GET /api/patients?token={{token}}"
```

**双 Tick 调度：**
- Critical 场景：60s 间隔
- Standard 场景：300s 间隔
- 结果暴露为 Prometheus 指标（`his_probe_success`, `his_probe_latency_ms`）

## 四、测试策略

| 层级 | 测试数 | 覆盖内容 |
|------|--------|---------|
| monitor-alert | 30 | 告警状态机、规则评估、静默匹配、通道格式化、通知分发 |
| monitor-collector | 17 | 指标解析、管道过滤/采样、模拟导出器 |
| monitor-processor | 20 | 窗口聚合、基线计算、四策略异常检测、流通道 |
| monitor-probe | 5 | 场景定义结构、URL 模板渲染、JSONPath 提取 |
| monitor-api | 2 | 中间件不阻塞请求 |
| monitor-core | 2 | 配置加载 |

**测试风格：**
- 纯单元测试，不依赖外部服务（Postgres/Redis 等）
- 异步测试使用 `#[tokio::test]`
- 对时间敏感的状态机测试直接构造 `DateTime` 控制时间流逝

**典型测试范式：**
```rust
#[tokio::test]
async fn test_pending_to_firing_after_duration() {
    let engine = RuleEngine::new(vec![rule_with_duration(1)]);
    let mut manager = AlertManager::new(engine, 10);
    manager.evaluate(&sample).await;
    assert_eq!(manager.active_alerts()[0].status, AlertStatus::Pending);
    tokio::time::sleep(Duration::from_secs(2)).await;  // 超过 duration_secs
    manager.evaluate(&sample).await;
    assert_eq!(manager.active_alerts()[0].status, AlertStatus::Firing);
}
```

## 五、Rust 惯用法亮点

### 5.1 workspace 组织

```toml
[workspace]
members = ["crates/monitor-*"]
[workspace.dependencies]
tokio = { version = "1", features = ["full"] }
# 所有 crate 共享版本，避免重复编译
```

### 5.2 trait 抽象

- `NotifyChannel` trait — 通知通道的可插拔接口
- `Exporter` trait — 基础设施导出器的统一契约
- 使用 `#[async_trait::async_trait]` 宏支持 async trait 方法

### 5.3 错误处理

```rust
// monitor-core/src/error/mod.rs
#[derive(Debug, thiserror::Error)]
pub enum MonitorError {
    #[error("configuration error: {0}")]
    Config(String),
    #[error("collector error: {0}")]
    Collector(String),
    // ...
}
```

`thiserror` 派生 `Display` + `Error` trait，配合 `anyhow` 在应用层使用。

### 5.4 并发安全

- `Arc<Mutex<T>>` 用于 shared mutable state
- `mpsc::channel` 用于事件驱动解耦
- `tokio::spawn` + `JoinHandle` 管理后台任务生命周期
- `#[derive(Clone)]` 让 `AppState` 可以在每个 Axum handler 中共享

### 5.5 配置分层

```yaml
# configs/monitor.yaml        — 默认值
# configs/monitor.dev.yaml     — 开发覆盖
# configs/monitor.prod.yaml    — 生产覆盖
```

`config-rs` crate 支持按环境合并配置，生产环境可集成 Nacos 配置中心热更新（架构预留）。

## 七、面试自述模板

> "HIS-Monitor-rs 是我独立设计并实现的 Rust 监控后端项目。它覆盖了从 Prometheus 指标采集、管道处理、时间窗口聚合、四种异常检测算法，到告警状态机、静默窗口、多渠道通知分发、REST API 和运营监控大屏的完整链路。
>
> 项目采用 workspace monorepo 组织，6 个 crate 分层解耦，通过 trait 抽象和 mpsc channel 实现各层之间的异步通信。核心的 AlertManager 是一个完整的有限状态机（PENDING→FIRING→ACKED→RESOLVED），通过事件驱动架构将规则评估和通知分发解耦到独立的 tokio task。
>
> 我特别重视测试，项目有 76 个单元测试覆盖了所有核心逻辑，包括状态机转换、指标解析、异常检测算法和白盒边界条件。
>
> 技术栈上，我选择了 Axum + Tokio 的异步 HTTP 框架，Tower 中间件体系做横切关注点，Prometheus 生态做指标暴露，所有外部存储通过 trait 抽象实现可替换性。"
